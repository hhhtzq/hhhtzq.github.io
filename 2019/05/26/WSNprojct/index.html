Simply WSN Application

Posted on 2019-05-26 | In WSN , contiki
Words count in article: | Reading time ≈

Introduction

In this application, the transmission is based on UDP (User Datagram Protocol), while the network is based on IPV6. The transmitter can read the sensor values of the external light sensor and the internal temperature sensor, and periodically transmit the acquired values to the receiving node that should display the data. In addition, every time the user button on the sending node is pressed, the receiving node should be notified that the button is pressed to represent the occupied state of the room. The generated data packet should include a counter, temperature and light value, the counter represents the number of data packets sent by the sender since startup. Finally, when the room is not occupied, each time the user button of the receiver is pressed, the LED on the sending node will switch once.

Photoresistor is a kind of resistor made by using the photoconductive effect of semiconductor to change the resistance value with the intensity of incident light, also known as photoconductive detector; incident light intensity, resistance decreases, incident light is weak, resistance Increase. There is another kind of weak incident light, the resistance decreases, the incident light intensity, the resistance increases.

Photoresistors are generally used for light measurement, light control, and photoelectric conversion (converting changes in light into changes in electricity). Commonly used photoresistors are cadmium sulfide photoresistors, which are made of semiconductor materials. The photoresistor’s sensitivity to light (that is, spectral characteristics) is very close to the human eye’s response to visible light (0.4~0.76) μm. As long as the human eye can sense the light, it will cause its resistance to change.

3️⃣ Protocols

  1. IPV6

Concepts:IPv6 is the abbreviation of English “Internet Protocol Version 6” , and is the next generation IP protocol designed by the Internet Engineering Task Force (IETF) to replace IPv4. The number of addresses can be claimed as every Grain sand with an address.

IPv6 is located at the third layer of the Internet protocol stack, the network layer. The data processed by layer 3 is data packets. The device that connects it to the Internet is a host or router. The host can choose a PC, a portable computer or a sensor board for sending and/or receiving data packets. The host is the source or destination of the packet. The router is responsible for packet forwarding and is responsible for selecting the next router to forward it to its final destination. The Internet consists of many interconnected routers. They receive data packets on one interface, and then use another interface to send data packets to another transit router as fast as possible.
The length of the IPv6 address is 128 bits, which is 4 times the length of the IPv4 address. So IPv4 dotted decimal format is no longer suitable for IPv6 address format, using hexadecimal representation. There are three ways to represent IPv6:

(a)Hexadecimal notation
The format is X:X:X:X:X:X:X:X, where each X represents 16b in the address, expressed in hexadecimal, for example: ABCD:EF01:2345:6789:ABCD:EF01:2345 :6789

(b)0-bit compressed representation
In some cases, an IPv6 address may contain a long period of 0s, which can be compressed into “::”. However, to ensure the uniqueness of address resolution, “::” can only appear once in the address, for example: FF01:0:0:0:0:0:0:1101 → FF01::1101

(c)Embedded IPv4 address notation
In order to achieve IPv4-IPv6 interworking, the IPv4 address will be embedded in the IPv6 address. At this time, the address is often expressed as: X:X:X:X:X:X:dddd, the first 96b is expressed in hex, and the last 32b The address is expressed in dotted decimal notation of IPv4, for example ::192.168.0.1

The IPv6 protocol mainly defines three types of addresses: Unicast Address, Multicast Address, and Anycast Address:

(a)Unicast address
used to uniquely identify an interface. Data packets sent to a unicast address will be transmitted to an interface identified by this address.

(b)Multicast address
used to identify a group of interfaces (usually this group of interfaces belong to different nodes). Data packets sent to the multicast address are transmitted to all the interfaces identified by this address.

(c)Anycast address
used to identify a group of interfaces (usually this group of interfaces belongs to different nodes). The data packet sent to the anycast address is transmitted to the one of the set of interfaces identified by this address that is closest to the source node (measured according to the routing protocol used).

Transport: It is a layer above the network layer, which provides additional functions for the network layer, such as retransmitting lost packets or ensuring that the order of sending and receiving packets is the same. This layer will show the application layer “ “Network Services” layer, they provide services for sending or receiving data. TCP and UDP are the most commonly used transmission protocols in the Internet.

  1. UDP

Concepts:User Datagram Protocol (UDP) is a simple transport layer protocol for connectionless, unreliable datagrams. In the TCP/IP model, UDP provides a simple method above the network layer and below the application layer The interface. UDP only provides unreliable delivery of data. Once it sends the data sent by the application to the network layer, it does not retain the data backup (so UDP is sometimes considered to be an unreliable datagram protocol).
Unlike TCP, UDP does not provide message fragmentation and reassembling at the other end, this means that the application must be able to make sure that the entire message has arrived and is in the right order.

Features:
① UDP is connectionless, that is, there is no need to establish a connection before sending data (of course, there is no connection to release at the end of sending data), so the overhead and delay before sending data are reduced;
② UDP uses best effort delivery, that is, reliable delivery is not guaranteed, so the host does not need to maintain a complicated connection status table;
③ UDP is oriented to the message;
④ UDP has no congestion control;
⑤ UDP supports one-to-one, one-to-many, many-to-one, and many-to-many interactive communication;

  • Network stack

I divided the function of implementing the project into four parts:

1) The light sensor and temperature sensor are connected to the transmitter. The sensor data, room occupancy and a counter data are stored in the structure message.
2) If the button is not pressed, it means that the room is not occupied. The transmitter will periodically send these data to the receiver every five seconds.
3) If the button is pressed, it means that the room is occupied, and the state of the button determines whether the data will be sent.
4) The button of the receiver can also request the occupancy status of the transmitter’s room. If the room is not occupied, then the receiver’s button will troggle the transmitter’s LED , otherwise it will not.

Transmitter

For the sender I have four “create”

1) UDP connection: First register a UDP connection and attach a callback function to it. For the incoming data packet, the callback function will be called, and then the UDP data packet will be sent to the specified IP address.
2) A structure and a pointer: Pack and store data such as sensor data and room occupancy status.
3) Receiver callback: It will be called by UDP library each time we receive a packet.
4) Sensor values’ function: Get the functions of the light sensor, temperature sensor and counter for easily use.

First of all, my transmitter will set periodic time , register the UDP connection and activate the light sensor. Then the program will enter a while (1) in an infinite loop, here is the default. In this infinite loop my transmitter has been working then, for the working state, one is that the button is not pressed, and the other is that the button is pressed.

Both of these working states will first create a link-local multicast address to all nodes.

For the working state where the button is not pressed, the transmitter will send data periodically every five seconds. In this working state, its LED will be troggled by the receiver’s button.

For the working state where the button is pressed, it will read the time when the button is pressed and released, and print the time, and only send data when the key is pressed.In this working state, its led will not be troggled by the receiver’s button.

After judging the working status, it will take sensor readings and store them into the message structure and then send a UDP packet to a specified IP address. Finally reset timer.

For the receiver, it has only one working mode, receiving the data packet of the sender, in addition, it can also query the occupancy status of the sender’s room through its buttons and callback functions.

(c)Scalability
If we place the wireless sensor nodes in multiple rooms and multiple floors, according to multicast method I use. First of all, it is a one-to-one group transmission mode, which is different from broadcasting, in the same subnet. Only the hosts participating in the multicast group will receive the packet, and the other hosts will not be interfered. Second, the packet will be forwarded to the hosts in the same group on another subnet through the router.Therefore, in terms of scalability, multicast requires clients of the same data stream to join the same group to share a data stream, saving the load of the server. Have the advantages of broadcasting.

But compared with unicast, multicast has no error correction mechanism in its protocol index, and it is difficult to remedy after packet loss and error. I think it can be replaced by a certain fault tolerance mechanism. The fault-tolerance mechanism can reduce the vulnerability of the system and make the system reliable and stable. For example, when nodes that seem to be unable to communicate are encountered: 1 stop communication directly 2 communicate only nodes that can communicate 3 communicate in another way 4 pseudo-communication.

Another issue to face is battery life. Based on the above calculations, in active mode, its working time is 1049 hours, about 43 days. This is a reasonable situation. This means that the battery needs to be replaced frequently. Therefore, for nodes of different work types, we need to use different work modes. In addition, we can also introduce a sensor sleep mechanism to reduce power consumption. In order to prevent sensor nodes from wasting energy when receiving unexpected data packets, we can consider the sleep mechanism. For example, coordinate strategies to determine which nodes should go to sleep.

In the case of more than multiple nodes, we can use a stump topology to connect these nodes. This method not only ensures a large network coverage, but also does not prevent the battery life from being too short, more flexible and efficient.
